# Agent 生命周期与自更新（Agent Lifecycle & Self-Update）

> 状态: stable
> 最后更新: generated-by-sdd-doc-update
> 反推来源: local workspace facts

## 设计意图

约束该领域的关键设计事实和可验证行为，供需求验收、实现计划和测试证据引用。

本文件描述长期有效的领域约束：只写必须满足什么、允许什么、禁止什么，以及如何验证；不写实现步骤、临时文件路径、局部命名方案、可替换技术选型或伪代码。若路径本身是稳定协议表面，必须说明其契约身份与验收方式。

## 约束空间

### 不变量（始终成立的断言）

| 编号 | 不变量 | 验收方式 |
|------|--------|----------|
| IU001 | Agent 注册到 Server 时必须携带可解析的版本标识；Server 必须持久化该版本。 | 注册帧包含版本字段，Server 侧 Agent 记录的 Version 字段非空。 |
| IU002 | Agent 版本标识必须单调递增（语义化版本或构建序号），Server 依据此标识判断新旧。 | 版本比较逻辑对合法版本格式返回确定的序关系。 |
| IU003 | Agent 自更新过程必须保证原子性：要么新二进制替换成功并重启，要么旧二进制继续运行；禁止处于中间不可用状态。 | 更新中断（网络断开、进程被杀）后，Agent 进程必须仍可启动并正常服务。 |
| IU004 | 更新或卸载操作必须可审计：操作发起方、目标 Agent、操作结果必须可追溯。 | Server 侧有操作日志或审计记录。 |

### 规则（允许 / 禁止）

| 编号 | 规则 | 允许/禁止 | 验收方式 |
|------|------|-----------|----------|
| RU001 | Server 必须能通过反向 tunnel 主动向指定 Agent 发起更新指令。 | 允许 | Server 调用更新 RPC 后，目标 Agent 开始下载新版本。 |
| RU002 | Agent 必须验证下载的二进制的完整性（校验和或签名）；校验失败必须拒绝替换并保留旧版本。 | 允许 | 下载被篡改的二进制后，Agent 不替换并上报校验失败。 |
| RU003 | Agent 更新必须由 Server 主动触发或 Agent 自主轮询触发；禁止第三方未授权触发。 | 禁止 | 无有效认证的更新请求被拒绝。 |
| RU004 | 卸载操作必须清理二进制、配置、数据和系统服务注册；禁止卸载后残留可执行文件。 | 禁止 | 卸载后 `/usr/sbin/collia`、`/etc/collia/`、服务注册均不存在。 |
| RU005 | 更新期间 Agent 必须先完成当前进行中的 RPC 调用或安全中断后才能退出；禁止更新导致进行中调用无响应。 | 禁止 | 更新时进行中的 terminal session 收到断开通知或优雅关闭。 |
| RU006 | 同一 Agent 不允许并发执行多个更新操作。 | 禁止 | 第二个更新请求被拒绝或排队，不会并行替换二进制。 |

## 输入 / 输出约束

### 输入合法性

- 合法输入：能被领域规则、公开 API 或人工确认事实支撑的约束。
- 非法输入：只描述实现步骤、具体库选择或不可验证愿望的内容。
- 处理语义：非法输入应该被改写为可验证约束，或从领域规范中移除。

### 输出约束

- 每条关键不变量或规则必须有稳定编号和验收方式。
- 输出内容必须能被需求、实现计划或测试证据引用。
- 输出不得降低已有 Domain Spec 的约束强度。

## 状态模型

| 状态 | 含义 |
|------|------|
| `draft` | 约束仍在形成，允许继续补充或收紧。 |
| `stable` | 约束已被实现、测试或需求引用，修改需要同步引用方。 |
| `deprecated` | 约束不再适用，但保留迁移说明和历史引用。 |

### 状态迁移

| 起始状态 | 事件 | 目标状态 | 允许/禁止 |
|----------|------|----------|-----------|
| `draft` | 约束通过评审且有验收方式 | `stable` | 允许 |
| `stable` | 领域语义被替代且有迁移路径 | `deprecated` | 允许 |
| `deprecated` | 直接恢复为稳定约束且无重新评审 | `stable` | 禁止 |

## 错误条件与失败语义

| 错误条件 | 可观测表现 | 是否可恢复 | 是否幂等 |
|----------|------------|------------|----------|
| 约束引用丢失 | 引用检查或评审失败 | 是，恢复缺失 ID 后重新校验 | 是 |
| 实现细节混入 Domain Spec | 评审标记为规范问题 | 是，改写为领域约束 | 是 |
| 状态语义不一致 | 测试或验收结果冲突 | 是，收紧状态模型后重测 | 是 |

## 结果约束（非功能性）

| 维度 | 约束 | 验收方式 |
|------|------|----------|
| 可追踪性 | 约束 ID 必须可被需求、实现计划和测试证据回挂 | 引用检查 |
| 确定性 | 相同输入状态必须产生相同领域判断 | 重复执行同一验收场景 |
| 一致性 | 领域状态、可观测行为和错误语义必须互相匹配 | 评审与回归测试 |
| 安全性 | 不记录密钥、Token 或不可公开配置值 | 文档审计 |
